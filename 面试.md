## 一、3 句话项目自我介绍（30 秒电梯版）

> **Slides Engine 是面向教学场景的「交互式课件编辑与播放引擎」**：编辑端基于 `Engine / Workspace / Operation` 内核做画布树编辑，预览端通过 `post-me` 接收同一份 `TreeNode` JSON 还原渲染，并支持课堂任务编排（红包、表扬榜、视频节）和 Cocos 游戏桥接。
>
> 我负责的核心点是：**低代码画布内核扩展**（Behavior 注册 + JSON Schema 属性面板）、**编辑端到预览端的 iframe 协议**、**撤销栈与快照节流**。
>
> 整体范式接近开源 Designable，但在它之上沉淀了一层**教学物料 + 跨端协议 + 微前端宿主适配**，让同一棵树可在编辑、独立预览页、微前端宿主、原生壳里一致播放。

## 二、2~3 分钟深入版（按这个顺序讲，最稳）

**1）业务背景**：教育课件编辑器 → 课堂播放，差异化是「不是静态 PPT，而是组件化 DSL」，每种物料都有独立 Behavior + Schema。

**2）总体架构**（口述时按分层描述）：
- 内核层 `packages/core`：`Engine → Workbench → Workspace → Operation → TreeNode`，无 React 依赖。
- React 绑定层 `packages/react`：`DesignerEngineContext`、observer 组件，仅消费引擎 API。
- 物料层 `editor/RegistryBehaviors`：把 Root/Card/Shape/Game 等业务组件挂到 `GlobalRegistry`。
- 端侧应用：`editor`（设计器）/ `task`（课堂端）/ `preview` / `bridge/*`（游戏壳）。

**3）三个最值得讲的设计点**：
- **Workspace = 一页课件**：内含两个 Viewport（主画布 + 大纲），共用 `MoveHelper`，但 `moveInsertionType` 不同（`all` vs `block`）—— 一套算法两种插入语义。
- **History 快照 + idle 合并**：`Operation.snapshot()` 用 `requestIdleCallback` 防抖，`History.locking` 防止 undo 过程中再次 push，`maxSize=100` 控内存。
- **跨 iframe 协议**：编辑端 `ParentHandshake` + 显式 `remoteOrigin`（不用 `*`），iframe `load` 后才建 Messenger，子端 `ChildHandshake` 注册 `getSlideData` → `dispatch.page.updatePage`。

**4）成果**：新增物料零改内核；编辑/预览/课堂三端协议统一；撤销流畅度可观测。

## 四、面试官最爱问的 10 个问题 + 应答骨架

> 答题统一公式：**结论 → 项目里怎么落 → 权衡 → 演进**。

### Q1 撤销为什么不用 Redux time-travel，而用快照？
> 画布是高频局部变更 + 深嵌套，用 `@slides/reactive` 细粒度 observable 性能更好；撤销靠 `History.serialize/from` 全量快照，与"整树 JSON 持久化"形态一致，实现简单。**代价**：单步 serialize 大课件偏重 → 演进路线是 Worker 序列化或操作日志 + 关键帧。

### Q2 `requestIdle` 合并快照会不会丢一步撤销？
> 极端情况会有一帧延迟。对**离散操作**（drop 结束、blur）走同步 push，对**连续输入**走 idle 合并，可两全。

### Q3 为什么是 `Engine → Workspace → Operation`？协作编辑挂哪？
> Workspace = 一页，独立视口、独立历史；Operation 承载 Tree + Selection + MoveHelper + AnimateEngine。**协作要挂在 Workspace 上**：把 mutation 打成 OperationLog 广播，或换 CRDT 文档层；不要塞进 React 组件。

### Q4 双 Viewport 不会事件冲突吗？
> 共享 Operation，但 DOM 容器和 `nodeIdAttrName`（画布 vs `outlineNodeIdAttrName`）不同；`MoveHelper` 通过 `activeViewport` + `moveInsertionType` 限制大纲只允许块级插入。

### Q5 拖拽插入位置怎么算？源节点和实例节点什么区别？
> `MoveHelper` 用指针 + 候选矩形算 `ClosestPosition`（Before/After/Inner）。物料栏是 `isSourceNode`，进画布时 `clone(parent)` 成实例；树内移动是 `remove + 改 parent`，依赖全局 `TreeNodes` Map 做 O(1) findById。

### Q6 iframe 预览安全怎么做？
> 用 `post-me` 的 Promise 化握手，`new URL(previewUrl).origin` 作为显式 `remoteOrigin`，不用 `*`；iframe `load` 后才建 Messenger 防止时序静默失败。**进一步**：payload 加 HMAC 或短期 token 换 JSON。

### Q7 同一渲染核要在微前端 + 独立预览两种宿主跑，怎么适配？
> `PostMessageClient` 构造时按 `player==='preview'` 分支：本地走 `BroadcastChannel('Signalling')`，宿主走 `window.microApp.forceDispatch`；外部协议统一为 `CoursewareCommand.TransferMessageReceive`。

### Q8 加新物料要改哪些层？
> 加 `createBehavior`（selector + designerProps + propsSchema）+ 编辑态/运行态组件 +（必要时）Setter。**不要动** Engine / TreeNode 事件总线 / History 语义。

### Q9 课件 schema 升级怎么兼容？
> 加 `schemaVersion` 字段，预览端"读旧写新"——未知字段忽略、缺省给默认值；保存时跑迁移脚本。

### Q10 任务系统和画布节点怎么绑定？
> 任务模型有 `elementId`，对应稳定节点 id。**复制页时维护映射表**更新引用，**保存前校验**任务引用的 id 仍存在。

## 一、先纠正一个常见误解

**项目不是"用 Designable"，而是"参考 Designable 范式"**——核心抽象 `Engine / Workspace / Operation / TreeNode` 的命名和分层与 Designable 高度一致，但代码在 `packages/core` 里是**二次开发后的内部版本**，加了教学场景的物料层、跨端协议、微前端适配。

> 面试时一定要这么说，否则会被追问"那你做了什么"而答不上来。

## 二、为什么不从零自研一个画布编辑器？

低代码画布要解决的是一组**已经被解决过 N 次**的通用问题：

| 通用问题 | 自研代价 |
|---|---|
| 可观察组件树 + 局部更新 | 设计 observable 粒度（ref/shallow/computed），踩 mobx/proxy 的坑 |
| 拖拽插入位置判定 | 实现 `ClosestPosition`、双视口共享算法、缩放/transform 坐标换算 |
| 撤销重做一致性 | 序列化快照 + locking 防递归 + idle 合并 |
| 选区 / Hover / 多选 / 键盘快捷键 | 事件总线、Driver 抽象 |
| 属性面板与组件解耦 | JSON Schema → 表单（Formily 系） |
| 设计器 UI 容器 | 插槽式布局、面板拖拽 |

这些**没有业务价值**，只有**复杂度**。从零做要 3~6 人月才能到能用，且 bug 难穷尽。**业务团队的核心价值在"教学物料 + 课堂播放"，不在重新发明编辑器**。

## 三、为什么是 Designable，而不是 GrapesJS / Craft.js / 自研 React-DnD？

| 候选 | 拒绝理由 |
|---|---|
| **GrapesJS** | 偏 Web 页面搭建（HTML/CSS 导出），输出不是 JSON 树而是 DOM 字符串，不适合**结构化 DSL** + 多端渲染 |
| **Craft.js** | 偏 React 内嵌编辑，组件强耦合 React，**难以做 headless 内核**给非 React 端复用 |
| **react-dnd 自研** | 只解决拖拽，没解决树/历史/Schema 表单 |
| **Designable** ✅ | 引擎与 React 解耦、Behavior 注册扩展、与 Formily 天然搭配做 Schema 表单、TreeNode 是纯 JSON 易跨端 |

**Designable 的关键命中点**：
1. **内核与 UI 解耦** → 后续可让 task / preview / 微前端宿主复用同一棵树。
2. **Behavior 注册模式** → 业务组件零改内核，符合教学物料频繁迭代的需求。
3. **JSON Schema 驱动属性面板** → 跟 Formily 配合，新增 Setter 不用写表单代码。
4. **TreeNode = 纯 ITreeNode JSON** → 跨 iframe / 跨端只传一个对象就够。

## 一、Setter 是什么？（一句话定位）

> **Setter = 属性面板里的"一格输入控件"**。
>
> 一个组件的 `propsSchema` 里每个字段都对应一个 Setter，比如：
> - `width: number` → `NumberPicker` Setter
> - `color: string` → `ColorInput` Setter
> - `align: 'left' | 'center' | 'right'` → `RadioGroup` Setter
>
> 在 Designable / Formily 体系里，**Schema 字段 + Setter 组件 + 表单值** 三者通过 `x-component` 绑定。

本项目里这层在 `packages/react-settings-form` 中，能看到 `CanvasSetter`、`BackgroundSetting` 等业务专用 Setter。

---

## 二、什么时候**不需要**新增 Setter（先排除）

如果属性是这些类型，**直接用内置就行，不要重复造**：

| 字段类型 | 用什么 |
|---|---|
| 数字 / 字符串 / 布尔 | `NumberPicker` / `Input` / `Switch` |
| 枚举（少量选项） | `Select` / `RadioGroup` |
| 颜色 | `ColorInput` |
| 简单数组 / 对象 | `ArrayItems` / `ObjectSetter` |
| 富文本 | 已有 `RichTextSetter` |

**判断标准**：能用 JSON Schema + 内置 `x-component` 描述的，**永远不要新增 Setter**。

---

## 三、什么时候**必须**新增 Setter（5 个真实场景）

### 场景 1：**复合数据结构 + 跨字段联动**
**例子**：背景设置 `BackgroundSetting`——同时管 `bgType`（颜色/图片/渐变）、`bgColor`、`bgImageUrl`、`bgRepeat`，**选了图片才显示重复方式**。
**为什么不用内置**：内置 Setter 是"一字段一控件"，跨字段联动要写在一个组件里才能共享 state。
**项目印证**：`packages/react-settings-form/BackgroundSetting`。

---

### 场景 2：**需要画布级实时预览的属性**
**例子**：`CanvasSetter`——拖动滑块改画布尺寸，**要实时反馈到画布**而不是等用户失焦保存。
**为什么不用内置**：内置 Setter 只管表单值，需要直接调 `engine.workbench.workspaces[i].operation` 同步画布的，必须自定义。

---

### 场景 3：**业务领域专用控件**
**例子**：
- **资源选择器**：从 CDN 资源库选图片/视频，需要打开内部 Modal、调内部 API、回填 URL
- **游戏选择器**：从游戏列表选 Cocos 包，回填 `gameId + version`
- **字体选择器**：从 `@slide/fonts` 列表选字体，且要 preview
- **音频/视频时长选择器**：选了 URL 后自动探测时长回填

**为什么不用内置**：业务 API + 业务 UI 强耦合，**没有通用版**可用。

---

### 场景 4：**所见即所得的可视化编辑控件**
**例子**：
- **Padding/Margin 四向输入器**（带可视化方框示意）
- **变换矩阵编辑器**（旋转/缩放手柄）
- **动画时间轴 Setter**（关键帧打点 UI）
- **形状路径编辑器**（贝塞尔锚点）

**为什么不用内置**：交互是图形化的，不是输入框能表达的。

---

### 场景 5：**绑定外部数据/状态的控件**
**例子**：
- **任务绑定 Setter**：选画布节点 → 关联到课堂任务的 `elementId`
- **节点引用 Setter**：从当前页其他节点里选一个作为依赖
- **国际化文案 Setter**：切语言看不同 locale 的输入框

**为什么不用内置**：值不是单纯的常量，而是**指向系统其他对象的引用**，需要自己做选择器 UI + 校验。

---

## 四、新增 Setter 的最小代价（5 步）

```ts
// 1. 写 Setter 组件
const MyCustomSetter = (props) => {
  const { value, onChange } = props
  return <div>...自定义 UI...</div>
}

// 2. 注册到 Formily 组件库
SchemaField.registerComponents({ MyCustomSetter })

// 3. 在 Behavior 的 propsSchema 里使用
propsSchema: {
  myField: {
    type: 'string',
    'x-component': 'MyCustomSetter',
    'x-component-props': { /* 配置 */ }
  }
}

// 4. （可选）国际化 messages
// 5. （可选）默认值 / 校验规则
```

**关键点**：不动 Engine、不动 TreeNode、不动 History。只是**给 Formily 加一个组件**而已。

## 五、面试官追问应对

### Q：什么时候不应该新增 Setter，而应该改组件 props 设计？
> 如果你为了一个字段写了 200 行 Setter，先反思 **props 是不是设计太奇怪**——比如把多个独立属性塞进一个对象。Setter 复杂度 ≈ props 复杂度，**优先简化 props**。

### Q：Setter 之间能否共享状态？
> 不行，每个 Setter 是隔离的。**跨字段联动必须写成一个复合 Setter**（场景 1），或在 Schema 上用 `x-reactions` 让 Formily 自动联动。

### Q：Setter 性能不好怎么办？
> Setter 在面板打开时才挂载，且选区切换会重渲。**重交互组件懒加载** + 用 `useMemo` 缓存非受控部分。

---

## 六、一句话总结（背下来）

> **新增 Setter = 字段太"业务"或太"图形化"，靠 JSON Schema + 内置控件描述不出来时**——比如背景设置、CDN 资源选择、动画时间轴、任务绑定。**评判标准是"业务耦合度"和"交互复杂度"，不是"想不想自定义 UI"**。

## 一、最准确的定义

> **Setter 本质就是一个普通的 React 组件**，只不过它要**符合表单控件的接口约定**：接收 `value` + `onChange`，用来编辑某个属性的值。

```tsx
// 这就是一个最简单的 Setter
const MySetter = ({ value, onChange }) => (
  <input value={value} onChange={e => onChange(e.target.value)} />
)
```

## 二、三种"新增 Setter"的程度（由轻到重）

### 程度 1：**直接用内置 Setter，零开发**
绝大多数场景。Schema 里写一行就行：
```ts
width: {
  type: 'number',
  'x-component': 'NumberPicker'  // 内置的，不用写组件
}
```
**这种情况你根本不需要"新增 Setter"。**

---

### 程度 2：**包装内置组件 + 业务参数（轻量级）**
比如想要一个"带 px 单位的数字输入框"：
```tsx
const PxInput = ({ value, onChange }) => (
  <NumberPicker value={value} onChange={onChange} addonAfter="px" />
)
```
**这是包装，不是从零写**——本质还是用内置组件，只是加了点业务皮肤。

---

### 程度 3：**完全自研一个新组件（重量级）**
当内置组件**根本表达不了**这个交互时，比如：
- 时间轴关键帧打点 UI
- 贝塞尔曲线锚点拖拽
- CDN 资源选择 Modal
- 四向 Padding 可视化输入器

这时候你才真正需要**从零写一个 React 组件**，但**接口仍然是 `value / onChange`**——这样它才能被 Formily 当 Setter 用。

## 三、关键认知：Setter 的"开发"和"使用"是两件事

| 角色 | 做什么 |
|---|---|
| **Setter 开发者** | 写一个符合 `value/onChange` 协议的 React 组件，注册到 Formily 组件库 |
| **物料开发者** | 在 `propsSchema` 里通过 `x-component: 'XxxSetter'` 引用它 |
| **课件设计者（最终用户）** | 在属性面板里看到这个控件，点点点编辑值 |

**你问的"自己开发"指的是第一层**，但日常 95% 的工作其实是第二层——**只用，不写**。

## 一、Formily 的 JSON Schema 在标准 JSON Schema 上扩展了什么？

标准 JSON Schema（draft-07）只有 `type / properties / required / enum` 等**数据校验**字段，**完全不管 UI**。Formily 在它之上加了一组 `x-*` 前缀字段，把"数据 + UI + 行为"塞进一个 Schema 里。

### Formily Schema 的核心字段分类

```ts
{
  type: 'string',           // ① 标准 JSON Schema：数据类型
  title: '宽度',             // ② 标签
  default: 100,             // ③ 默认值
  enum: [...],              // ④ 校验/枚举

  // ===== Formily 扩展（x-* 前缀） =====
  'x-decorator': 'FormItem',          // 包裹层（Label + 错误提示）
  'x-component': 'NumberPicker',      // 真正的 Setter 组件名
  'x-component-props': { addonAfter: 'px' },  // 传给 Setter 的 props
  'x-validator': [{ required: true }],         // 校验规则
  'x-display': 'visible',             // 显示状态：visible/hidden/none
  'x-pattern': 'editable',            // 交互状态：editable/disabled/readPretty
  'x-reactions': [...],               // 联动逻辑（最关键！）
  'x-content': '...',                 // 静态内容
}
```

**设计精髓**：把 UI 行为通过 `x-` 命名空间嫁接到 Schema 上，**协议层面与 JSON Schema 标准兼容**，又能描述完整的表单 UI。

---

## 二、Schema 是如何变成 UI 的？（渲染管道）

```
JSON Schema
    ↓ Schema.compile()  解析 + 编译表达式
Field 对象（响应式 model）
    ↓ <SchemaField />   按字段递归渲染
React 组件树
    ↓ x-decorator + x-component
最终 DOM
```

**关键对象**：
- **`Form`**：整个表单的状态容器，存所有 `values`、`errors`、`fields` 引用
- **`Field`**：每个字段对应一个响应式 model，自带 `value / errors / display / pattern` 等可观察字段
- **`SchemaField`**：递归读 Schema，给每个字段实例化 Field，绑到 `x-component`

**为什么用响应式而不是 Redux**：表单字段量大、字段独立变更频繁，**细粒度 observable**（基于 `@formily/reactive`）让"改一个字段只重渲一个字段"，这是 Formily 性能的根本。

## 三、表单联动的 4 种方案（重点）

### 方案 1：**`x-reactions` —— 声明式联动（最推荐）**

这是 Formily 解决联动问题的**杀手锏**。

#### 例 1：A 字段值决定 B 字段是否显示
```ts
{
  type: { type: 'string', enum: ['color', 'image'] },
  imageUrl: {
    type: 'string',
    'x-component': 'ImagePicker',
    'x-reactions': {
      dependencies: ['type'],          // 依赖 type 字段
      fulfill: {
        state: {
          visible: '{{$deps[0] === "image"}}'  // 表达式
        }
      }
    }
  }
}
```

#### 例 2：A 字段变化触发 B 重新计算值
```ts
{
  width: { type: 'number' },
  height: { type: 'number' },
  area: {
    type: 'number',
    'x-reactions': {
      dependencies: ['width', 'height'],
      fulfill: {
        state: {
          value: '{{$deps[0] * $deps[1]}}'
        }
      }
    }
  }
}
```

#### 例 3：异步联动（请求依赖项）
```ts
{
  province: { 'x-component': 'Select' },
  city: {
    'x-component': 'Select',
    'x-reactions': [
      {
        dependencies: ['province'],
        fulfill: {
          run: `
            field.loading = true
            fetch('/cities?p=' + $deps[0])
              .then(r => r.json())
              .then(list => {
                field.dataSource = list
                field.loading = false
              })
          `
        }
      }
    ]
  }
}
```

**`x-reactions` 三大能力**：
| 能力 | 字段 | 作用 |
|---|---|---|
| 状态联动 | `state.visible / display / pattern / value` | 根据依赖改 UI 状态 |
| 行为联动 | `run` | 跑一段命令式代码（异步、副作用） |
| 模式匹配 | `when` | 条件成立才触发 |

---

### 方案 2：**`x-validator` —— 校验联动**
跨字段校验，比如「确认密码必须等于密码」：
```ts
confirmPwd: {
  'x-validator': {
    validator: '{{(val, rule, ctx) => val === ctx.form.values.password ? "" : "密码不一致"}}'
  }
}
```

---

### 方案 3：**Effects API —— 命令式联动（处理复杂逻辑）**
当 `x-reactions` 表达式写不下时，在外面用 `useFormEffects`：

```tsx
const Form = () => {
  return (
    <FormProvider form={form}>
      <FormConsumer>
        {() => {
          form.addEffects('linkage', () => {
            onFieldValueChange('type', (field) => {
              const target = form.query('imageUrl').take()
              target?.setState({ visible: field.value === 'image' })
            })
          })
        }}
      </FormConsumer>
    </FormProvider>
  )
}
```

**用 Effects 的场景**：
- 多字段联动逻辑很复杂，写在 Schema 里太长
- 需要访问 React 外部状态（Redux / Context）
- 需要保留可调试的断点

---

### 方案 4：**Computed Field —— 派生字段**
某字段始终是其他字段的纯函数：
```ts
fullName: {
  'x-reactions': {
    fulfill: {
      state: { value: '{{$values.firstName + $values.lastName}}' }
    }
  }
}
```
本质是 `x-reactions` 的语法糖。

## 四、为什么 Formily 的联动设计**优于** AntD Form？

| 维度 | AntD Form | Formily |
|---|---|---|
| 联动写在哪 | React 组件 JSX 里手写 `Form.useWatch + setFieldsValue` | Schema 里声明 `x-reactions` |
| 改字段重渲范围 | Form 整体重渲（开了 `dependencies` 才精细） | **字段级响应式**，只重渲变化字段 |
| 跨字段校验 | 自己写逻辑 | `x-validator` 直接拿 ctx |
| 异步联动 | 自己写 useEffect | `x-reactions.run` 内置 |
| 配置化 | 难——逻辑藏在 React 代码里 | 易——纯 JSON 可序列化、可后端下发 |

**最大价值**：**联动逻辑是数据，不是代码**——可以从后端动态下发、可以存数据库、可以被低代码平台二次配置。

## 二、本项目的联动落点：4 种真实做法

### 做法 1：**Field 组件的 `reactions` prop（命令式联动）** ⭐ 主力方案

证据：[DisplayStyleSetter/index.tsx](file:///Users/zouyu/Desktop/slides-engine/packages/react-settings-form/src/components/DisplayStyleSetter/index.tsx)

### 做法 2：**复合 Setter 内部消化联动**（黑盒化）

证据：[PolyInput/index.tsx](file:///Users/zouyu/Desktop/slides-engine/packages/react-settings-form/src/components/PolyInput/index.tsx)、[BoxTransformStyle](file:///Users/zouyu/Desktop/slides-engine/packages/react-settings-form/src/components/BoxTransformStyle/index.tsx)、[CornerInput](file:///Users/zouyu/Desktop/slides-engine/packages/react-settings-form/src/components/CornerInput/index.tsx)、[PositionInput](file:///Users/zouyu/Desktop/slides-engine/packages/react-settings-form/src/components/PositionInput/index.tsx)

把多个字段塞进一个 Setter 组件，对外只暴露一个聚合值，内部用 `useState + useEffect` 自己管联动：


**典型场景**：
- **背景设置**：颜色 / 图片 / 渐变三选一 → 组件内部切渲染
- **变换矩阵**：translate/rotate/scale 多字段联动 → 一个 Setter 内部维护
- **边框**：4 个方向 × (宽度/颜色/样式) → BorderStyleSetter 一组件管完

**这种做法的好处**：把 Schema 变简单（外层只看到一个字段），联动复杂度被封装进 Setter，**对物料开发者透明**。

### 做法 3：**Setter 内部用 `field.value` + observer 做响应式联动**

证据：[CanvasSetter](file:///Users/zouyu/Desktop/slides-engine/packages/react-settings-form/src/components/CanvasSetter/index.tsx)、[BoxTransformStyle](file:///Users/zouyu/Desktop/slides-engine/packages/react-settings-form/src/components/BoxTransformStyle/index.tsx)

```tsx
import { useField, observer } from '@slides/react'

const BoxTransformStyle = observer((props) => {
  const field = useField()
  // field 是响应式 model，field.value / field.parent.value 任意访问
  // 整个组件被 observer 包裹，依赖自动收集
})
```

**原理**：Formily 的 `Field` 是响应式对象，**配合 `observer` 自动追踪依赖**，写法和 mobx 一致——比 React 的 `useState + useEffect` 更省事。

### 做法 4：**联动跨出表单的边界 → 走 Engine API**

这是本项目特有的、**Formily 标准方案覆盖不到**的场景。

比如改了画布尺寸 → 要同步到 `engine.workbench.workspaces[i].operation`：


**关键**：联动的对象是**画布树（TreeNode）**，不是另一个表单字段。这种情况**只能在 Setter 组件里命令式调引擎 API**，Schema 层无法表达。

## 四、为什么本项目这样选？（面试可主动抛）

> 1. **物料属性大多独立**：宽高、透明度、自动播放——彼此无依赖，没必要写 `x-reactions`。
>
> 2. **复杂联动天然适合复合 Setter**：背景、边框、变换这类强耦合属性，**封装在一个 React 组件里比拆成多个字段 + 表达式更可维护**——逻辑一处可读、可断点。
>
> 3. **联动对象常常是画布**：`x-reactions` 是表单内字段间联动，但课件设计器更多是「改属性 → 影响画布」**跨边界场景**，必须靠 Setter 命令式调 Engine API。
>
> 4. **可调试性**：JSX 里的 `reactions={f => ...}` 可以打断点；Schema 表达式字符串调试体验差，遇到问题难定位。

## 一、案例 A：`BoxTransformStyle` —— 一个字段拆成 3 个子控件

### 1. 它要解决的问题

CSS `transform` 是一个字符串：`translate(10px, 20px) rotate(45deg)`。
但用户在面板里希望分开操作 **X 偏移 / Y 偏移 / 旋转角** 三个独立控件。

**矛盾**：底层数据是 1 个字段，UI 是 3 个控件。

### 2. 怎么写（核心代码 4 步）

```tsx
export const BoxTransformStyle = observer((props) => {
  const field = useField()                                  // ① 拿到 Formily field
  const [rotateValue, setRotateValue] = useState<any>()     // ② rotate 用本地 state（性能优化）

  // ③ 解析：从字符串里抽出 x / y / rotate
  const translateMatch = (value) => value.match(/translate\((-?\d+)px, (-?\d+)px\)/) || []

  // ④ 反向拼接：3 个子控件改任一个，都拼回完整 transform 字符串
  const createTransformHandler = (transType, props) => ({
    value: 解析出对应分量,
    onChange(v) {
      let valueTrans = props.value
      if (transType === 'translateX') valueTrans = valueTrans.replace(/translate\(...\)/, `translate(${v}, ${y}px)`)
      // ...rotate / translateY 同理
      props.onChange?.(valueTrans)   // ← 关键：调用父级 onChange，把字符串吐回去
    }
  })

  return (
    <SizeInput {...createTransformHandler('translateX', props)} />
    <SizeInput {...createTransformHandler('translateY', props)} />
    <Slider value={rotateValue} onChange={onChange} />   // ← rotate 走 state
  )
})
```

### 3. 为什么可以这么写

**关键在于 Formily 的 Setter 协议**：

```
Setter 拿到的不是某个具体字段，而是一对协议参数：{ value, onChange }
```

这意味着：
- **value** 不一定是单个原子值，可以是任意 JSON（字符串、对象、数组）
- **onChange** 的语义是"我把整个 value 写回去"，不规定中间形式
- 因此 Setter 内部可以**任意拆解 value**、**任意组装回去**，外面看不见

这就是为什么"一个字段对应一个 UI 控件"是误解——**真正的协议是"一个字段对应一个 Setter"**，Setter 内部想长什么样、有几个子控件，全是它自己的事。

### 4. 数据怎么流（最重要）

```
                          ┌─── 编辑用户拖 X 滑块 ───┐
                          ▼                          │
   TreeNode.props.style.transform                     │
       = "translate(10px, 20px) rotate(45deg)"        │
                          ▼                           │
            ① Formily 把 props.style.transform 作为 value 注入 Setter
                          ▼                           │
            ② BoxTransformStyle 用正则解析出 x=10 y=20 rotate=45
                          ▼                           │
            ③ 渲染成 3 个子控件，分别 value=10 / 20 / 45
                          ▼                           │
                ④ 用户改 X 子控件 ────────────────────┘
                          ▼
            ⑤ createTransformHandler('translateX').onChange(newX)
                          ▼
            ⑥ 内部正则 replace 拼回完整字符串
                          ▼
            ⑦ props.onChange(`translate(${newX}px, 20px) rotate(45deg)`)
                          ▼
            ⑧ Formily 写入 form.values.style.transform
                          ▼
            ⑨ SettingsForm 监听 values 变化 → setProps 到 TreeNode
                          ▼
            ⑩ TreeNode 是 observable → 画布该节点重渲
```

**两个边界细节**：
- **rotate 用 useState 而非每次重算**：滑块拖动是高频，避免重复正则解析
- **props.value 变化时 useEffect 同步本地 state**：保证撤销/重做时 UI 也跟着回滚

## 二、案例 B：`BorderStyleSetter` —— 多字段共享一个"位置选择器"

### 1. 它要解决的问题

边框有 5 个位置（center / top / right / bottom / left），每个位置有 3 个属性（style / width / color），共 **15 个字段**。
UI 上只想显示当前选中位置那 3 个，其它隐藏。

**矛盾**：15 个真实字段 vs 1 个"虚拟选择器" + 3 个可见控件。

### 2. 怎么写（核心代码）

```tsx
export const BorderStyleSetter = observer(() => {
  const field = useField<FieldType>()

  // ① 创建一个独立的响应式状态：当前激活位置
  const currentPosition = useMemo(
    () => observable({ value: parseInitPosition(field) }),
    [field.value]
  )

  // ② 关键：reactions 工厂，给每个 Field 决定显隐
  const createReaction = (position) => (field) => {
    field.display = currentPosition.value === position ? 'visible' : 'hidden'
    //              ↑ 这里读 currentPosition.value，会被自动收集为依赖
  }

  return (
    Positions.map(position =>
      <>
        <Field name={`border-${position}-style`} reactions={createReaction(position)} ... />
        <Field name={`border-${position}-width`} reactions={createReaction(position)} ... />
        <Field name={`border-${position}-color`} reactions={createReaction(position)} ... />
      </>
    )
  )
})
```

### 3. 为什么可以这么写

**3 个底层能力凑在一起**：

| 能力 | 出处 | 作用 |
|---|---|---|
| `observable({ value })` | `@slides/reactive` | 创建一个**外部响应式变量**，独立于表单字段 |
| `<Field reactions={fn}>` | Formily | reactions 是一个**会被自动追踪依赖**的函数，依赖变化时重跑 |
| `field.display = ...` | Formily Field model | 直接改 field 的 model 属性，UI 自动响应 |

**核心技巧**：`currentPosition` 不是表单字段（不会被 serialize），而是一个**临时 UI 状态**——但它依然是响应式的，所以 reactions 函数会在它变化时被重新执行。

> 这相当于自己造了一个**"轻量 x-reactions"**：依赖收集是自动的，但写法是 React 友好的。

### 4. 数据怎么流

```
TreeNode.props.style 里实际存了 15 个字段（borderTopStyle, borderTopWidth, ...）
                        ▼
       Formily 把它们都映射成 Field model（响应式）
                        ▼
              组件挂载，parseInitPosition 扫描 15 个字段哪个有值
                        ▼
              用扫描结果初始化 currentPosition.value = 'top'
                        ▼
              渲染 5×3 = 15 个 <Field>，每个绑 createReaction(position)
                        ▼
   reactions 函数被 Formily 调用，访问 currentPosition.value 触发依赖收集
                        ▼
              结果：position === 'top' 的 3 个字段 visible，其它 hidden

   ─── 用户切换位置（暂时被注释了，但逻辑保留）───
                        ▼
              currentPosition.value = 'right'
                        ▼
       响应式系统检测到变化 → 通知所有依赖它的 reactions 重跑
                        ▼
       15 个字段重新计算 display → DOM 切换可见性
                        ▼
                       （表单值没动，只是 UI 显隐改了）

   ─── 用户改 right-color ───
                        ▼
              ColorInput.onChange → Field 写值
                        ▼
              form.values.style.borderRightColor 更新
                        ▼
              SettingsForm useEffect 监听 → setProps 到 TreeNode
                        ▼
              画布节点 borderRightColor 应用
```

## 三、统一抽象：本项目 Setter 的"数据流四层"

```
┌────────────────────────────────────────────────────────┐
│  Layer 4: TreeNode.props  ←─ 真相（写到 JSON、能 serialize）
└────────────────┬───────────────────────────────────────┘
                 │  Engine API（setProps）
┌────────────────▼───────────────────────────────────────┐
│  Layer 3: Formily Form / Field  ←─ 响应式状态层
└────────────────┬───────────────────────────────────────┘
                 │  value / onChange
┌────────────────▼───────────────────────────────────────┐
│  Layer 2: Setter 组件  ←─ 业务层，可以拆/拼/联动
│   - 内部 useState（高频性能）
│   - 内部 observable（UI 临时状态）
└────────────────┬───────────────────────────────────────┘
                 │  React 渲染
┌────────────────▼───────────────────────────────────────┐
│  Layer 1: 子控件（SizeInput / Slider / ColorInput）
└────────────────────────────────────────────────────────┘
```

**关键规则**：
1. **真相只有一份**：永远在 Layer 4（TreeNode）。
2. **Layer 3 是镜像**：Formily 把 props 反射成可观察的 Form values。
3. **Layer 2 可自由发挥**：Setter 想怎么拆字段、怎么加临时状态都行，但**最终必须通过 onChange 把完整 value 写回**。
4. **回流靠 SettingsForm 统一桥接**：在 [SettingsForm.tsx](file:///Users/zouyu/Desktop/slides-engine/packages/react-settings-form/src/SettingsForm.tsx) 里 `useEffect` 监听 form values，调引擎 `node.setProps()`。

## 四、为什么这种"Setter 自治"模型是合理的

| 取舍 | 收益 |
|---|---|
| Schema 不写联动 | Schema 保持可序列化、可后端下发的纯净 |
| 联动塞进 Setter | 复杂业务逻辑可调试、可断点、可写复杂正则/异步 |
| Setter 内部用 observable | 临时 UI 状态不污染表单值，**不会被序列化进课件 JSON** |
| Form values 是镜像不是真相 | 撤销/重做改 TreeNode → Form 自动同步，无需手动 reset |

**用一句话总结**：
> **Setter 是"数据真相"和"用户交互"之间的翻译官**——它把真实的 props 字段拆成用户友好的 UI 形态（如 transform 拆 3 段、border 按位置分组），又把用户操作翻译回原始字段格式写回去。Formily 提供 `value/onChange` 协议作为契约，`@slides/reactive` 提供 UI 临时状态的响应式能力，两者结合让 Setter 既能复杂又能保持数据流单向清晰。

## 五、面试可直接背的版本

> 我们项目里复杂联动**走 Setter 自治模型**，以 `BoxTransformStyle` 为例：底层 `transform` 是单字段字符串，Setter 内部用正则拆成 X/Y/旋转三个子控件，每个子控件改值后再正则拼回完整字符串通过 `props.onChange` 写回——**对外仍是单字段协议，对内做了任意拆解**。`BorderStyleSetter` 更进一步，用 `@slides/reactive` 的 `observable` 造一个"虚拟位置选择器"作为 UI 临时状态，配合 Formily Field 的 `reactions` prop 让 15 个字段按位置自动显隐——**响应式依赖收集是自动的，但临时状态不进表单 values，也不会被 serialize 进课件 JSON**。整个数据流是 `TreeNode.props → Formily Form → Setter → 子控件`，写回时反向走，Setter 只需保证 `onChange` 吐出符合原字段语义的完整 value。

1. **真相只有一份**：永远在 Layer 4（TreeNode）。
2. **Layer 3 是镜像**：Formily 把 props 反射成可观察的 Form values。
3. **Layer 2 可自由发挥**：Setter 想怎么拆字段、怎么加临时状态都行，但**最终必须通过 onChange 把完整 value 写回**。
4. **回流靠 SettingsForm 统一桥接**：在 [SettingsForm.tsx](file:///Users/zouyu/Desktop/slides-engine/packages/react-settings-form/src/SettingsForm.tsx) 里 `useEffect` 监听 form values，调引擎 `node.setProps()`。

## 四、为什么这种"Setter 自治"模型是合理的

| 取舍 | 收益 |
|---|---|
| Schema 不写联动 | Schema 保持可序列化、可后端下发的纯净 |
| 联动塞进 Setter | 复杂业务逻辑可调试、可断点、可写复杂正则/异步 |
| Setter 内部用 observable | 临时 UI 状态不污染表单值，**不会被序列化进课件 JSON** |
| Form values 是镜像不是真相 | 撤销/重做改 TreeNode → Form 自动同步，无需手动 reset |

**用一句话总结**：
> **Setter 是"数据真相"和"用户交互"之间的翻译官**——它把真实的 props 字段拆成用户友好的 UI 形态（如 transform 拆 3 段、border 按位置分组），又把用户操作翻译回原始字段格式写回去。Formily 提供 `value/onChange` 协议作为契约，`@slides/reactive` 提供 UI 临时状态的响应式能力，两者结合让 Setter 既能复杂又能保持数据流单向清晰。

## 五、面试可直接背的版本

> 我们项目里复杂联动**走 Setter 自治模型**，以 `BoxTransformStyle` 为例：底层 `transform` 是单字段字符串，Setter 内部用正则拆成 X/Y/旋转三个子控件，每个子控件改值后再正则拼回完整字符串通过 `props.onChange` 写回——**对外仍是单字段协议，对内做了任意拆解**。`BorderStyleSetter` 更进一步，用 `@slides/reactive` 的 `observable` 造一个"虚拟位置选择器"作为 UI 临时状态，配合 Formily Field 的 `reactions` prop 让 15 个字段按位置自动显隐——**响应式依赖收集是自动的，但临时状态不进表单 values，也不会被 serialize 进课件 JSON**。整个数据流是 `TreeNode.props → Formily Form → Setter → 子控件`，写回时反向走，Setter 只需保证 `onChange` 吐出符合原字段语义的完整 value。
