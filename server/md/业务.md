你要学会的不是“背表”，而是能把这条链路讲成一个完整后端开发故事：

```text
Slide 课件
  -> Page 页面
    -> PageResourceRelation / Resource 资源
    -> CourseTask 互动任务
```

你可以按这 8 步准备。

**1. 需求拆解**

你要能说清楚这个业务在干什么：

> 这个服务端是给课件编辑器用的。一个课件 `Slide` 下面有多个页面 `Page`，每个页面保存一棵组件树 JSON。页面里可能引用图片、音频、视频等资源，所以需要 `Resource` 和页面资源关系表。页面中的某些元素还可能绑定课堂互动任务，所以需要 `CourseTask`，通过 `pageId + elementId` 找到某个页面元素对应的任务。

你要会回答：

- 为什么课件要拆成 Slide 和 Page？
- 为什么页面内容用 JSON 存？
- 为什么资源不直接塞进页面 JSON？
- 为什么任务要单独建表？
- `elementId` 是干什么的？

核心表达：

> 页面结构灵活，用 JSON 存；资源和任务需要查询、复用、关联，所以单独建表。

**2. 数据建模**

你要重点会这几个关系：

```text
Slide 1 - N Page
Page N - N Resource，通过 PageResourceRelation
Page 1 - N CourseTask
CourseTask 通过 pageId + elementId 绑定页面里的某个组件
```

每张表你要知道作用：

| 表 | 作用 |
|---|---|
| `Slide` | 课件主体，记录状态、结构、锁信息 |
| `Page` | 课件页面，保存页面组件树 |
| `Resource` | 资源元数据，比如文件名、MD5、路径、类型 |
| `PageResourceRelation` | 页面和资源的关联关系 |
| `CourseTask` | 页面元素绑定的互动任务 |

你要会讲索引和约束：

```prisma
@@index([slideId, sortIndex])
@@unique([pageId, resourceId])
@@index([pageId, elementId])
@@index([slideId, taskType])
```

面试表达：

> 索引是按查询场景加的。页面经常按 slideId 查并按 sortIndex 排序；资源关系要避免重复绑定，所以 pageId + resourceId 做联合唯一；任务经常按 pageId + elementId 查询，所以加联合索引。

**3. 接口设计**

你要会讲常见接口怎么设计：

```text
POST /slides/create
GET /slides/:slideId
POST /slides/:slideId/save
POST /slides/pages/create
POST /slides/pages/:pageId/save
POST /slides/pages/course-tasks/save
GET /slides/pages/:pageId/course-tasks
POST /slides/:slideId/publish
POST /slides/:slideId/go-to-edit
POST /slides/:slideId/exit-edit
```

你要知道每类接口干什么：

- 创建课件：生成 `Slide`
- 创建页面：生成 `Page`，挂到某个 `Slide`
- 保存课件：保存课件整体结构
- 保存页面：保存页面组件树，同时更新资源引用
- 保存任务：保存页面上的互动任务
- 获取课件：返回 Slide + Page + Resource
- 发布课件：修改状态
- 编辑锁：防止多人同时编辑

核心表达：

> 接口不是随便拆的，而是按前端编辑器动作拆。用户创建课件、添加页面、编辑页面、绑定资源、保存任务、发布课件，每个动作对应一个后端接口。

**4. 鉴权**

你要会讲这三种角色：

```text
viewer：查看
editor：编辑
admin：发布、管理
```

你项目里的权限逻辑：

- 查看课件：`viewer/editor/admin`
- 创建、保存页面：`editor/admin`
- 发布课件：`admin`
- 登录后通过 JWT 访问接口
- `RolesGuard` 判断角色

面试表达：

> 前端隐藏按钮只是体验，真正的权限必须在服务端接口层做。比如普通 viewer 即使手动调用保存接口，也会被 RolesGuard 拦住。

**5. 业务实现**

你要按具体业务讲。

创建页面：

```text
1. 校验 slideId 对应课件是否存在
2. 生成 pageId
3. 根据 pageType 创建默认页面结构
4. sortIndex 放到当前页面最后
5. 写入 Page 表
```

保存页面：

```text
1. 接收 mainContentStructure JSON 字符串
2. parse 成 JSON
3. 更新 Page.mainContentStructure
4. 如果传了 fileMd5List，更新页面和资源的关联关系
5. 返回 pageId
```

保存任务：

```text
1. 接收 slideId、pageId、courseTaskList
2. 软删除旧任务
3. 批量创建新任务
4. 任务里保存 elementId、taskType、payload
```

获取课件：

```text
1. 查询 Slide
2. include pages
3. include page.resourceRelations.resource
4. 按 sortIndex 排序
5. 转换成前端需要的 DTO
```

核心表达：

> Service 负责业务流程，Repository 负责 Prisma 查询。比如保存页面不是单纯 update page，还要同步资源关系，所以业务逻辑放在 Service/Repository 中处理。

**6. 数据一致性**

这是最重要的后端味道。

你要会讲三个点：

第一，事务：

> 保存页面时，同时更新页面内容和资源关系，必须放事务里。否则页面保存成功、资源关系失败，会出现数据不一致。

第二，唯一约束：

> 页面和资源关系用了 `@@unique([pageId, resourceId])`，防止同一个页面重复绑定同一个资源。

第三，软删除：

> 删除页面、任务、资源关系时不直接物理删除，而是设置 `deletedAt`。这样方便恢复和保留历史。

再加一个并发点：

> 课件编辑用 `currentLockToken` 做基础编辑锁。进入编辑生成 token，退出编辑校验 token。后续可以加锁过期时间，或者加 version 乐观锁防止覆盖保存。

**8. 部署运维**

你要会讲上线前要做什么：

```text
1. 配置环境变量 DATABASE_URL、JWT_SECRET、CORS_ORIGIN
2. 构建服务 npm run build
3. 执行 prisma migrate deploy
4. 启动服务 npm run start:prod
5. 检查日志
6. 检查 Swagger 或健康接口
7. 确认数据库表结构和种子数据
```

注意区分：

```text
开发环境：prisma migrate dev
生产环境：prisma migrate deploy
```

面试表达：

> 生产环境不会临时生成 migration，而是执行已经提交到代码仓库里的 migration，保证数据库结构可追踪、可回滚、团队一致。

你最后可以把整条链路背成这一段：

> 这个项目里，`Slide` 是课件主体，一个课件下有多个 `Page`。`Page` 保存页面组件树 JSON，因为编辑器结构灵活，不适合完全拆表。页面中引用的图片、音视频等资源放到 `Resource` 表，页面和资源通过 `PageResourceRelation` 建多对多关系，并用 `pageId + resourceId` 做联合唯一，避免重复绑定。页面里的互动题、课堂任务放到 `CourseTask`，通过 `slideId + pageId + elementId` 绑定到具体页面元素。保存页面时，我会在事务里更新页面内容和资源关系，保证数据一致性。接口层用 JWT + RBAC 控制权限，viewer 只能看，editor 能编辑，admin 能发布。开发联调通过 Swagger，数据库结构通过 Prisma schema 和 migration 管理，上线时执行 migrate deploy。

你把这段说顺，基本就能让面试官感觉：你不是只看过代码，而是知道一个服务端功能怎么从需求落到数据库、接口、权限和上线。
