**正常服务端开发流程，你要会讲这个**

1. 先看前端需要什么数据  
比如编辑器需要创建课件、保存课件结构、保存页面内容、保存资源列表、保存课堂任务。后端第一步不是写代码，而是明确“接口契约”：请求字段、返回字段、错误情况、权限要求。

2. 设计数据模型  
你这里的核心表是：

- `User`：用户、密码哈希、角色
- `Slide`：课件主体，状态包括 `editing / published / packed`
- `Page`：课件页面，一页对应一棵组件树 JSON
- `Resource`：图片、视频、音频等资源
- `PageResourceRelation`：页面和资源的关联关系
- `Course / LessonBinding`：课程和课件绑定
- `CourseTask`：课堂任务，和页面元素 `elementId` 绑定

面试时你可以说：  
“课件编辑器的数据不是简单文本，而是结构化 JSON，所以我把课件主体、页面、资源、任务拆开存。页面内容用 JSON 字段保存组件树，资源和任务单独建表，方便查询、复用和软删除。”

3. 写接口 DTO 和校验  
你项目里用了 `class-validator` 和全局 `ValidationPipe`。这个很重要。

你可以说：  
“Controller 不直接相信前端传参，所有 body/query 都先经过 DTO 校验。比如 `pageType` 要转成 number 并校验整数，`fileMd5List` 要校验为字符串数组。全局开启 `whitelist` 和 `forbidNonWhitelisted`，避免多余字段进入业务层。”

4. Controller 只做路由，Service 做业务，Repository 做数据库  
这是后端面试很爱问的分层。

你可以这样讲：

- `Controller`：处理路由、参数、权限装饰器
- `Service`：处理业务规则，比如发布、锁定、保存任务
- `Repository`：封装 Prisma 查询，避免数据库逻辑散落

比如 `SlidesController` 接收 `/classroom-slides/slides/:slideId/save`，然后交给 `SlidesService.saveSlide()`，最终由 `SlidesRepository.updateSlideStructure()` 写数据库。

5. 鉴权和权限控制  
你这里有两层：

- `JwtAuthGuard`：校验用户是否登录
- `RolesGuard`：校验用户角色是否允许访问

角色包括：

- `viewer`：只能查看
- `editor`：可以编辑
- `admin`：可以发布、取消发布等高级操作

面试时这样说很像真实开发过：

“不是所有接口都只要登录就行，比如查看课件 viewer/editor/admin 都可以，但创建页面、保存页面需要 editor/admin，发布课件只允许 admin。我用 Guard + 自定义 `@Roles()` 装饰器做接口级权限控制。”

6. 处理数据一致性  
你项目里 `savePage`、`replaceTasks`、`appendTasks` 用了 Prisma transaction。

这个要重点准备，因为这是后端味道很重的点。

你可以说：

“保存页面时，不只是更新页面 JSON，还要同步页面引用的资源列表。如果先更新页面成功，资源关联失败，就会出现数据不一致。所以我把页面更新、旧资源关系软删除、新资源关系 upsert 放到一个事务里。”

任务保存也一样：

“保存任务是先软删除旧任务，再批量创建新任务，所以也必须放事务里，保证页面任务列表要么整体成功，要么整体失败。”

7. 软删除  
你的模型里大量用了 `deletedAt`。

可以这样讲：

“业务数据不直接物理删除，而是通过 `deletedAt` 做软删除。这样可以避免误删后无法恢复，也方便保留历史数据。查询时统一加 `deletedAt: null`，比如查课件、页面、资源、任务都只查未删除数据。”

8. 统一响应和异常  
你这里有：

- `ResponseInterceptor`：正常响应统一包成 `{ code, message, data }`
- `HttpExceptionFilter`：异常统一转换成 `{ code, message, data: null }`

面试表达：

“为了让前端处理接口更稳定，我做了统一响应格式。业务层只需要抛 `NotFoundException / ForbiddenException / UnauthorizedException`，最终由全局异常过滤器统一输出。”

9. Swagger 和联调  
你可以说：

“服务启动后会生成 Swagger，前端和后端可以基于接口文档联调。登录后拿 accessToken，在 Swagger Authorize 里配置 Bearer Token，就能测试需要鉴权的接口。”

**最容易被追问的点，你要特别补**

你可以优先补这几个：

- `Prisma transaction`：为什么保存页面要事务
- `JWT + refreshToken`：登录、刷新、退出流程
- `RBAC`：不同角色能访问哪些接口
- `deletedAt`：软删除的查询约束
- `JSON 字段`：为什么页面结构不拆成很多表
- `接口契约`：为什么保留 `/classroom-slides` 前缀，和前端保持兼容
- `锁机制`：`goToEdit / exitEdit` 为什么要 `lockToken`

一句很加分的话：

> 我不是把前端 JSON 原样一存就完了，而是把页面结构、资源引用、任务引用拆成不同模型；页面内容允许灵活 JSON，资源和任务独立建表，这样既兼容编辑器复杂结构，又能支持查询、复用、软删除和后续扩展。

你按这个方向准备，比单纯背 NestJS 八股强很多。
