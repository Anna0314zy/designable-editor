现在服务端的主链路是：

1. 先有课程 `Course`
   - 表：`Course`
   - 主键：`id`
   - 这个 `id` 就是前端/课程系统里的 `productId`
   - 本地没有真实课程中心，所以服务端会用 `ensureProduct(productId)` 自动兜底创建一个示例课程。

2. 再有课次绑定位 `LessonBinding`
   - 表：`LessonBinding`
   - 核心字段：
     - `mainId`：课程 id，也就是 `Course.id`
     - `serialNumber`：第几讲，比如 `1`、`2`、`IntroductoryLesson`
     - `slideId`：绑定到哪个课件
   - 唯一键是：`mainId + serialNumber`
   - 所以它表示：“某个课程的某一讲”。

3. 创建课件 `Slide`
   - 前端点“创建课件”时先调：
     - `POST /classroom-slides/slides/create`
   - 服务端只创建一条 `Slide`，返回：
     - `{ slideId }`
   - 这一步还没有和课程发生关系。

4. 绑定课程课次和课件
   - 拿到 `slideId` 后，前端马上调：
     - `POST /classroom-slides/lesson-packages/bind-slide`
   - 参数大概是：
     ```ts
     {
       mainId: productId,
       serialNumber: '1',
       slideId,
       slideTitle
     }
     ```
   - 服务端执行 `lessonBinding.upsert`
   - 如果这个课程这讲已经存在，就更新 `slideId`
   - 如果不存在，就创建一条 `LessonBinding`
   - 绑定关系就是：
     ```text
     Course.id = LessonBinding.mainId
     LessonBinding.slideId = Slide.id
     LessonBinding.serialNumber = 第几讲
     ```

5. 进入编辑器后创建页面 `Page`
   - 调：
     - `POST /classroom-slides/slides/pages/create`
   - 参数里带 `slideId`
   - 页面表 `Page.slideId` 指向 `Slide.id`
   - 也就是：一个课件下面有多页。

6. 保存课件整体结构和页面内容
   - 保存课件结构：
     - `POST /classroom-slides/slides/:slideId/save`
     - 存到 `Slide.slideStructure`
   - 保存单页内容：
     - `POST /classroom-slides/slides/pages/:pageId/save`
     - 存到 `Page.mainContentStructure`

7. 资源和任务再挂到页面/课件下面
   - 资源：
     - `Resource`
     - `PageResourceRelation`
     - 通过 `pageId + resourceId` 绑定资源到页面
   - 任务：
     - `CourseTask`
     - 字段里同时有 `slideId` 和 `pageId`
     - 所以任务属于某个课件里的某一页。

一句话总结：

```text
Course 课程
  -> LessonBinding 课程第几讲
    -> Slide 课件
      -> Page 页面
        -> Resource / CourseTask 资源和任务
```

绑定课程的关键不是 `Slide` 直接存 `courseId`，而是中间表 `LessonBinding`。它用 `mainId + serialNumber` 找到“课程的某一讲”，再把这一讲的 `slideId` 指向具体课件。

Slide 课件
  -> Page 页面
    -> PageResourceRelation / Resource 资源
    -> CourseTask 互动任务
