# Slides Engine — 工程化能力分析

从 **可维护性、扩展性、容错性、系统设计** 四维度，结合本仓库实际。

---

## 1. 可维护性

**优势**：

- **清晰分层**：`packages/core` 无业务 UI，问题定位可区分「引擎 bug」与「物料 bug」。
- **行为集中注册**：`editor/src/RegistryBehaviors.ts` 单文件可总览全量物料，利于 Code Review。
- **TypeScript 全链**：`ITreeNode`、`IEngineProps` 等约束跨包契约。

**债项**：

- 根 `package.json` 的 `workspaces` 与 `pnpm-workspace.yaml` 不完全一致（根级列了 `slides/*` 等而 yaml 更窄），**新人克隆后需以 pnpm 实际解析为准**。
- 子包 `name` 与目录名（如 editor 的 `play`）易混淆，**文档与脚本需统一**。

**建议**：在 `editor` 的 `package.json` 将 `name` 与业务称呼对齐，或显式在 README 说明别名。

---

## 2. 扩展性

**水平扩展（新组件）**：

- 增加 `createBehavior` + 实现编辑态/运行态组件 + 若需则扩展 `genPropsSchema`。
- 无需改 `Engine` 类核心逻辑。

**垂直扩展（新端）**：

- 新宿主只要实现 **树 JSON 渲染** 与 **消息适配**（参考 `PostMessageClient` 双分支），即可接入。

**协议扩展**：

- `pageType`、`props` 内新增字段需 **版本化**；建议保存时带 `schemaVersion` 字段（若尚未有，属演进点）。

---

## 3. 容错性

**已有**：

- `react-error-boundary` 包在根依赖中；`App` 中 Thumbnail/Animation 等使用 `WithErrorBoundary` 变体，**单面板崩不拖死全页**（需确认各入口均包裹）。

**风险**：

- 跨窗口 **postMessage** 无统一错误码回传；子窗口解析失败时依赖 `try/catch` 与 store 状态（`task` 的 `getSlideData` 应显式错误提示与重试策略）。
- `PostMessageClient` 与 microApp 强耦合，**宿主未注入**时运行期才暴露问题 —— 宜在启动时 feature-detect。

**建议**：

- 握手增加 **version 协商**；失败时父页 Toast + 关闭 iframe。
- 对 `JSON.parse` 与 `updatePage` 加 **Zod/JSON Schema 校验**。

---

## 4. 系统设计能力（从仓库反推团队要求）

- **多宿主思维**：同一渲染核，微前端 / 独立页 / iframe 多形态，说明对**部署环境异构**有现实压力。
- **教学业务深度**：任务类型、游戏桥、动效引擎，要求**领域模型**与**通用编辑内核**分离，避免污染 core。

---

## 5. 研发流程与质量门禁

**已见**：

- `husky` + `commitlint`（根 `prepare`），规范提交信息。
- ESLint + TypeScript 在子包中配置。

**可增强**：

- **CI 矩阵**：`pnpm -r build`、核心包单测、typecheck。
- **契约测试**：`serialize` 样例与预览端 `updatePage` 对拍。

---

## 6. 总结表

| 维度 | 评级（相对成熟中台） | 依据 |
|------|----------------------|------|
| 模块边界 | 优 | core / react / editor / common 清晰 |
| 扩展成本 | 优 | Behavior 注册模式 |
| 可观测性 | 中 | 大量 console，缺统一 trace id |
| 发布与版本 | 中 | workspace 协议依赖需文档化 |
| 容错 | 中-良 | 有 ErrorBoundary，跨端错误待体系化 |
