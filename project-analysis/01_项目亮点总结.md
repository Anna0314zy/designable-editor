# Slides Engine — 项目亮点总结

> 本文档基于仓库实际代码与目录结构归纳，面向**架构复述、面试表达与简历提炼**。

---

## 1. 业务价值（解决什么问题）

| 维度 | 说明 |
|------|------|
| **核心场景** | 面向教学场景的**交互式课件 / 幻灯片**生产与播放：编辑端产出结构化页面树，预览端按同一套渲染协议还原；并延伸至**课堂任务编排**（红包、游戏、表扬榜、视频节等）与**原生桥接播放**（如 Cocos 游戏容器）。 |
| **业务闭环** | **编辑（editor）→ 序列化树 → 预览 / 课堂端（task、preview、bridge）**：同一 `TreeNode` 语义在不同宿主中以不同壳呈现（设计器 iframe、独立预览页、微前端宿主）。 |
| **差异化** | 不是单纯静态 PPT，而是**组件化课件 DSL**：根节点、卡片、图形、富文本、音视频、摄像头、游戏嵌入等均有独立 **Behavior + JSON Schema 属性面板**，可扩展教学物料类型。 |

**工程沉淀价值**：沉淀了一套可复用的**低代码式画布内核**（Designable 演进形态）+ **教学业务插件层**（RegistryBehaviors、任务模型、跨端协议），后续新业务（题库页、互动测验）可挂载在同一引擎上。

---

## 2. 核心技术亮点（一句话版）

1. **引擎内核与 UI 解耦**：`Engine → Workbench → Workspace → Operation` 管理树、选区、拖拽、历史，React 仅做壳与订阅（`@editor/react`）。
2. **可扩展组件协议**：`GlobalRegistry` + `createBehavior`，按 `selector` 聚合设计期属性、拖拽规则、Schema 驱动的设置表单。
3. **双视口 + 插入语义**：主画布 `Viewport` 与大纲 `outline` 共用同一套 `MoveHelper` / `ClosestPosition` 算法，区分 `moveInsertionType`（如大纲仅 block 级）。
4. **历史栈与快照节流**：`History` 序列化 `Operation`，`snapshot` 通过 `requestIdle` 合并，避免每次按键入栈。
5. **跨边界数据传输**：编辑端与预览 iframe 使用 **post-me** 双向握手（`ParentHandshake` / `ChildHandshake`），显式 `remoteOrigin` 降低任意 origin 风险。
6. **运行时双形态动画**：`Operation` 挂载 `AnimateEngine`，编辑与预览侧有 `PreviewAnimateEngine` 等分化，支撑课件动效与课堂节奏。
7. **任务子系统（task）**：Rematch 管理当前页、全局资源与 `globalProps`（含 CDN、文件列表、首帧样式隐藏等），与教学产品中「环节 + 任务」模型对齐。

## 3. 核心模块闭环（设计思路 → 业务含义 → 实现要点）

### 3.1 `Engine` / `Workspace` / `Operation`

- **设计思路**：单一引擎实例挂全局事件；每个「页面」对应一个 `Workspace`，内含独立 `Operation`（一棵树 + 选区 + 变换状态）。
- **业务含义**：多页课件 = 多 workspace，`serialize()` 产出可持久化 / 可预览的结构。
- **实现要点**：`Engine.setCurrentTree` / `getCurrentTree` 委托当前 workspace 的 `operation.tree`；`Workspace.serialize` 将 `pageType` 与 `tree` 一并输出（与预览 store 对齐）。

### 3.2 `TreeNode` + 变更事件

- **设计思路**：节点树既可变又需可观察；所有结构变更走统一事件（Insert/Remove/UpdateProps…），便于联动撤销与外部订阅。
- **业务含义**：画布上每一个组件实例对应一个节点，props 即课件 JSON。
- **实现要点**：`TreeNodes` Map 全局索引；`designerProps` 由 `GlobalRegistry.getDesignerBehaviors` 归约；拖拽来源节点与操作树节点通过 `isSourceNode` 等区分。

### 3.3 `History` + `snapshot`

- **设计思路**：命令式改树 + 快照式撤销：每次有意义变更后 `serialize()` 入栈。
- **业务含义**：教师误操作可撤销，且栈大小有上限（默认 100）控制内存。
- **实现要点**：`locking` 防止恢复过程中再次 push；`snapshot` 内 `cancelIdle` + `requestIdle` 防抖。

### 3.4 编辑应用 `editor`（package 名 `play`）

- **设计思路**：启动时 `createDesigner()` 单例 + `RegistryBehaviors` 副作用导入注册物料。
- **业务含义**：装配具体课件组件集（Root、Card、Shape、Game…）。
- **实现要点**：预览打开 Modal iframe，`load` 后 `WindowMessenger` + `getSlideData` 推送 `workbench.workspaces` 序列化结果。

### 3.5 预览 `task` / `preview`

- **设计思路**：子窗口作为 **Handshake 子端**，收到 JSON 后 `dispatch.page.updatePage` 灌入 Redux 状态。
- **业务含义**：与编辑端解耦部署，仅依赖协议字段（`slideId`、`pageId`、`pages`、`fileList`）。
- **实现要点**：生产环境对 `VITE_PLAY_SERVER` 做 origin 归一化，避免错误 remoteOrigin。

## 4. 边界与异常（口述时可强调）

- **Handshake 时序**：必须 iframe `load` 后再建 Messenger，否则 `remoteHandle` 无效。
- **多引擎上下文**：`Designer` 要求 React 树里只有一个 `DesignerEngineContext`，否则抛错，防止状态串台。
- **历史与快照**：`history.locking` 或 workspace 缺失时 `snapshot` 直接 return，避免脏栈。
- **微前端与预览**：`PostMessageClient` 在 preview 模式用 `BroadcastChannel`，宿主模式走 `window.microApp`，分支错误会导致消息丢了无声失败。

## 5. 与「AI 项目」边界说明

本仓库**主体是传统可视化编辑 + 教学播放**，不包含 LLM/RAG/Agent 编排实现。AI 相关扩展见 `07_AI化扩展方向.md`；若面试被问到 RAG，应诚实界定范围并迁移到「若在本项目中引入 AI」的设计。

## 6. 简历可用药点（短句）

- 负责教学课件 **DSL + 低代码画布引擎** 扩展：基于 **Behavior 注册表 + JSON Schema** 实现多物料的设计态与属性面板统一。
- 打通 **编辑端 iframe 预览协议**（post-me 握手 + origin 约束），支撑课件跨部署预览与课堂端一致渲染。
- 优化 **撤销栈与快照**：idle 合并 + 历史上限，平衡交互流畅度与内存。
