# Slides Engine — 深度追问与标准回答（满分口述逻辑）

> 「标准回答」指：**结论 → 本项目怎么做 → 权衡 → 可演进**。可按 STAR 压缩。

---

## Q1：为什么内核不用 Redux，而用 reactive TreeNode？

**追问动机**：考察全局状态方案选型。

**标准回答**：

- **结论**：画布树是 **高频局部变更 + 深度嵌套**，适合细粒度响应式；Redux 适合可追溯的业务流水型状态。
- **本项目**：`TreeNode` 用 observable 标注字段，`designerProps` 用 computed 从 `GlobalRegistry` 归约，与 Designable 一脉的 **节点即状态** 模型一致。
- **权衡**：时间旅行调试不如 Redux DevTools 直观，但可通过 **History 快照** 覆盖教学场景的撤销需求。
- **演进**：若要做多人协作 OT/CRDT，需在 TreeNode 变更层插 **操作变换**，与是否 Redux 无必然绑定。

## Q2：`operation.snapshot` 为什么放在 `requestIdle` 里？

**标准回答**：

- **问题**：每次属性微调或拖拽中间状态都 `history.push` 会导致 **主线程序列化风暴**。
- **做法**：`cancelIdle` 取消上一次 pending，`requestIdle` 在空闲时 push，合并Burst操作。
- **边界**：若用户极快执行「改 → 立刻 Ctrl+Z」，理论上有 **一帧级** 延迟；若产品不接受，可对 **离散命令**（拖放结束、blur）走同步 push，对连续 input 走 idle。
- **本项目印证**：`History.locking` 在 undo 过程关闭快照，避免重入。

## Q3：双 Viewport 会不会重复绑定事件冲突？

**标准回答**：

- **设计**：同一 `Workspace` 共享 `Operation`，但 **两个 Viewport** 不同 DOM 容器与 `nodeIdAttrName`（画布 vs 大纲 `outlineNodeIdAttrName`）。
- **隔离**：`MoveHelper` 里通过 **当前 activeViewport**、插入类型限制大纲行为。
- **冲突防护**：事件派发带 `IEngineContext`，handler 能区分来源 viewport；若仍有交叉，需在 Driver 层过滤目标节点是否属于当前激活视口（需结合具体 Driver 实现口述）。

---

## Q4：iframe 预览协议如果被中间人篡改怎么办？

**标准回答**：

- **现状**：post-me 限定 **`remoteOrigin`**，避免任意窗口冒充子 frame。
- **不足**：同源策略下仍有 XSS 风险；数据完整性未内置签名。
- **演进**：敏感课件可对 payload **HMAC / JWT**，子窗口校验发行方；或使用 **短期一次性 token** 换 JSON。
- **边界**：本项目当前为 **教学课件非金融**，威胁模型以误配置与 XSS 为主。

---

## Q5：为什么 History 存在 Operation 上而不是 Engine 上？

**标准回答**：

- **语义**：撤销粒度是 **当前页**；切 workspace 应切换历史栈（每 `Workspace` 构造时 `new History`）。
- **若放 Engine**：需自行维护「当前页索引 → 历史实例」映射，复杂度上升。
- **印证**：`serialize` 来自 `operation.serialize()`，与栈条目一一对应。

---

## Q6：全局 `TreeNodes` Map 会不会内存泄漏？

**标准回答**：

- **用途**：O(1) `findById`，拖拽与外部 API 频繁查询。
- **风险**：长会话 + 大量创建销毁节点时 Map 只增不减会泄漏。
- **缓解**：页面卸载 `engine.unmount`、销毁 workspace 时遍历移除；或弱引用策略（需改实现）。
- **面试加分**：提出 **定期校验 Map.size vs 树节点数** 的调试断言。

---

## Q7：物料扩展时 `propsSchema` 与运行时渲染组件如何对齐？

**标准回答**：

- **约束**：`RegistryBehaviors` 里 `propsSchema` 决定设置面板；实际渲染由 `@slide/render-components` 等消费同一套 props key。
- **治理**：通过 **单一真相**（如共享 typings 包 `@editor/typing`）或生成器 `genPropsSchema` 减少漂移。
- **异常**：Schema 有字段但渲染组件未实现 → 运行时忽略或告警；应在 CI 做 **Schema vs 组件 props** 静态对齐检查（当前可作为改进点）。

---

## Q8：任务类型几十种，如何避免 store 膨胀？

**标准回答**：

- **现状**：Rematch `page` model 专注课件页；任务类型定义在 README/常量层，属于 **领域配置**。
- **原则**：任务状态机放独立 model 或服务端编排；前端只保留 **当前环节所需字段**。
- **演进**：任务插件化（按 `taskType` 懒加载配置模块）。

---

## Q9：若要把内核开源给第三方，你会先拆什么？

**标准回答**：

- **拆**：`@editor/core` + `@editor/react` 纯设计器；去掉 `@slide/*` 业务依赖（如 `AnimateEngine` 引用可改为可选注入）。
- **文档**：序列化 schema 版本、Behavior API、握手协议。
- **测试**：TreeNode mutation、History、MoveHelper 纯单测。

---

## Q10（开放）：和 Figma 本地撤销有什么区别？

**标准回答**：

- Figma 类 **图形引擎** 多用自定义文档模型 + 增量；我们是 **组件树 DSL**，快照实现简单。
- 若节点数上万，我们会转向 **增量 patch**；教学课件通常未到该量级。
