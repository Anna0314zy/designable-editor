# Slides Engine — 技术难点拆解

每条均包含：**难点本质 → 本项目解法 → 选型理由 → 边界与异常**。

---

## 1. 可变树 + 撤销重做的一致性

**难点**：既要高性能增量更新（拖拽、改属性），又要任意时刻可恢复到历史快照；若撤销过程中触发副作用再次入栈，会产生分叉或脏数据。

**解法**：

- `History` 使用 **serialize / from** 模式：`push` 时保存 `context.serialize()`（对 `Workspace` 即 operation 的树 + selected）；`undo/redo` 时 `locking = true` 调用 `from`，禁止递归 push。
- `Operation.snapshot()` 用 **`requestIdleCallback` 防抖**，并在 history `locking` 或缺失 workspace 时直接退出。

**选型理由**：快照比操作日志实现成本低，与现有「整树 JSON」持久化形态一致；教学课件单页节点规模可控，100 条历史内可接受。

**边界**：树特别大时需评估序列化耗时；可改为结构化克隆 + 差异补丁（当前未做）。

## 2. 拖拽插入位置的几何判定（MoveHelper + ClosestPosition）

**难点**：画布上多个容器嵌套时，需判断「前插 / 后插 / 内嵌 / 禁止」等，且大纲与主画布插入语义不同。

**解法**：

- `MoveHelper` 维护 `closestNode`、`viewportClosestDirection`、`ClosestPosition` 枚举（Before/After/Inner/…）。
- `Workspace` 内 **两个 Viewport**：主画布 `moveInsertionType: 'all'`，大纲 `moveInsertionType: 'block'`、`moveSensitive: false`，共享同一套 helper 逻辑但通过配置收窄行为。

**选型理由**：枚举化位置状态便于单元测试与调试；双视口配置避免写两套拖拽算法。

**边界**：极端缩放、transform 叠加时 offsetRect 与视觉不一致需要 Viewport 坐标换算一致（需结合 `TransformHelper` 与 DOM 测量核对）。

---

## 3. TreeNode 身份与来源节点（Source vs Operation）

**难点**：左侧物料栏拖入的「源节点」与画布上「实例节点」生命周期不同；拖拽中途可能 clone、挂父、从 Map 注册。

**解法**：`resetNodesParent` / `isSourceNode` 分支：源节点进画布时 `clone(parent)`；已在操作树内的节点移动时 `remove` + `shallowReset` 等，保证 **TreeNodes Map** 与父子关系同步。

**选型理由**：避免源组件污染全局树，同时支持「拖到画布才实例化」的低代码范式。

**边界**：`TreeNode.findById` 全局 Map —— 需注意测试或 HMR 重复注册导致的泄漏（生产依赖页面卸载时清理引擎）。

---

## 4. iframe 预览的安全与时序

**难点**：`postMessage` 任意 origin 易被劫持；iframe 未 load 完调用子窗口会静默失败。

**解法**：

- `new URL(previewUrl).origin` 作为 **`remoteOrigin`** 传入 `WindowMessenger`。
- **监听 iframe `load`** 后再 `ParentHandshake`，再 `remoteHandle().call('getSlideData', data)`。
- 子页 `task` 侧 `ChildHandshake(messenger, methods)` 注册 `getSlideData`，解析后 `dispatch.page.updatePage`。

**选型理由**：post-me 提供 Promise 化握手，比裸 postMessage 更易组合；显式 origin 满足安全基线。

**边界**：`previewUrl` 与父页不同部署域时需两边 CORS/证书一致；`destroyOnClose` 清空 iframe 避免旧连接残留。

## 5. 微前端宿主下的消息双路径（PostMessageClient）

**难点**：同一渲染内核要在「独立预览页」和「微前端子应用」两种宿主运行。

**解法**：构造函数按 `player === 'preview'` 分支 **BroadcastChannel 'Signalling'** vs **`window.microApp.forceDispatch`**；主端监听 `CoursewareCommand.TransferMessageReceive`。

**选型理由**：教学场景可能有多窗口广播；宿主侧统一走微前端数据通道。

**边界**：`console.log` 较多（现状），生产应降级；监听器未 remove 可能重复订阅。

---

## 6. 动画引擎与操作模型的耦合

**难点**：动效触发依赖节点 id、时间轴、课堂事件；不能与纯 UI 动画混为一谈。

**解法**：`Operation` 构造时 `new AnimateEngine()`；`common/animate` 提供 **编辑态** 与 **PreviewAnimateEngine**（预览侧 triggerMap、自动播放等）。

**选型理由**：动效作为课件状态的一部分，挂在 Operation 上便于与选区、序列化策略统一讨论（若未来需要序列化动画配置，扩展 `serialize` 即可）。

**边界**：预览引擎文件含调试 `console.log`，且与内核循环依赖风险需关注打包树摇。

---

## 7. 任务域模型与课件页的关联

**难点**：课堂「任务」类型多（游戏、红包、视频…），需与 slide 页、元素 id 关联。

**解法**：`task/README` 中定义的 `ITask` 含 `elementId`、`taskType`、扩展 `taskExt`；与 `PageType`（normal/game/video）等枚举配合。

**选型理由**：任务与课件解耦为独立 DSL，便于运营配置不必改内核。

**边界**：需保证编辑端生成的 `elementId` 与任务引用一致，否则课堂端找不到绑定节点。

---

## 8. 小结表

| 难点 | 核心机制 | 异常防护 |
|------|----------|----------|
| 撤销一致 | locking + 快照 | idle 合并、maxSize |
| 拖拽命中 | ClosestPosition + 双视口 | 缩放/transform 对齐 |
| 节点生命周期 | Source clone / Map | 卸载清理 |
| 跨 iframe | post-me + load | origin 显式 |
| 双宿主消息 | Channel vs microApp | 避免重复监听 |
