# Slides Engine — 架构设计分析

---

## 1. 总体拓扑

```
┌─────────────────────────────────────────────────────────────────┐
│                        应用层 (Apps)                              │
│  editor (设计器)  │  task (课堂/任务端)  │  preview  │  bridge/*   │
└──────────────┬──────────────────┬────────────────┬───────────────┘
               │                  │                │
┌──────────────▼──────────┐  ┌────▼─────┐  ┌──────▼──────────────┐
│ @editor/react           │  │ Rematch  │  │ post-me / postMessage│
│ Designer / Panels / UI  │  │ store    │  │ 跨窗口/微前端        │
└──────────────┬──────────┘  └──────────┘  └─────────────────────┘
               │
┌──────────────▼──────────────────────────────────────────────────┐
│ @editor/core (引擎内核)                                            │
│ Engine → Workbench → Workspace → Operation                       │
│   ├─ TreeNode (可观察树 + 全局 Map 索引)                          │
│   ├─ Selection / Hover / MoveHelper / TransformHelper            │
│   ├─ History (序列化快照)                                        │
│   └─ Viewport x2 (画布 + 大纲)                                   │
└──────────────┬──────────────────────────────────────────────────┘
               │
┌──────────────▼──────────────────────────────────────────────────┐
│ 物料与渲染公共层 (common/*)                                        │
│ slide-editor / render-core / render-components / slide-shape     │
│ animate (AnimateEngine / PreviewAnimateEngine)                    │
└──────────────────────────────────────────────────────────────────┘
```

**分层思想**：

- **内核层**：与 React 无关，可测、可复用到其他 UI 框架或 headless 自动化。
- **React 绑定层**：`DesignerEngineContext`、observer 组件、面板组合，只消费引擎 API。
- **业务物料层**：`editor/RegistryBehaviors` 将具体组件与 Schema、国际化、拖拽策略绑定到 `GlobalRegistry`。
- **端侧应用层**：同样协议（树 JSON）下，编辑、预览、任务、桥接各自实现数据入口与壳工程。

## 2. 单页工作区模型（核心抽象）

`Workspace` 是「一页课件」的完整上下文：

| 组成 | 职责 |
|------|------|
| `viewport` | 主画布：真实拖拽、碰撞检测敏感（`moveSensitive: true`） |
| `outline` | 大纲树视口：结构级插入（`moveInsertionType: 'block'`） |
| `operation` | 当前页树 + 选区 + 移动/变换辅助 |
| `history` | 绑定到本页 `Workspace` 的撤销栈（`History<Workspace>` 的 context 实际序列化 `Operation` 经 `serialize`） |

`getEventContext()` 把 `workbench, workspace, engine, viewport` 打成一个包，供事件管道里做**多 workspace 协调**（例如 `getAllSelectedNodes` 遍历所有 workspace）。

**数据流（编辑态）**：

1. 用户操作 DOM → Driver 转自定义 Event → `workspace.dispatch` → `Engine.dispatch`。
2. Mutation 类事件修改 `TreeNode`，触发响应式更新。
3. 关键路径调用 `operation.snapshot()` → idle 后 `history.push`。
4. 持久化：`workspace.serialize()` → API 保存；预览：多 workspace `map(serialize)` 经 postMessage 发出。

## 3. 树与注册表（扩展性设计）

- **树模型**：`ITreeNode` 是跨端传输的纯数据；`TreeNode` 是运行期对象，带 `parent/root/operation` 等环。
- **GlobalRegistry**：`getDesignerBehaviors(node)` 对节点上所有匹配 `selector` 的 `IBehavior` 做 **reduce**，合并 `designerProps`、locales。新组件只需 `createBehavior` + `setDesignerBehaviors` 数组一项。

**设计亮点**：行为可 **extends**（类型上支持组合），属性 Schema 用 `genPropsSchema` 等工厂生成，避免手写大量重复 Formily/表单配置。

## 4. Monorepo 与依赖边界

`pnpm-workspace.yaml` 收录：`packages/*`、`editor`、`common/*`、`preview`、`task`、`bridge/*`。

- **`packages/core`**：引擎，依赖 `@editor/shared`、`@slides/json-schema`（类型）、`@slides/reactive`。
- **`packages/react`**：壳组件，依赖 core。
- **`editor`**：聚合入口；引入 `@slide/fonts` PWA、antd、`RegistryBehaviors`。
- **`common/animate`**：被 `Operation` 直接 `new AnimateEngine()` 引用，体现**动画与操作模型紧耦合**（动效作用在节点时间轴/触发器上）。

**边界注意**：`packages/react-settings-form` 提供 CanvasSetter、BackgroundSetting 等与课件业务强相关的 Setter，属于「表单层公共」，而非纯内核。

## 5. 跨端与宿主集成

| 路径 | 机制 | 用途 |
|------|------|------|
| editor → iframe 预览 | `post-me` WindowMessenger + `ParentHandshake` | 推送序列化 pages |
| task Preview | `ChildHandshake`，子页调 `getSlideData` | 接收父窗口数据灌 store |
| render-core `PostMessageClient` | microApp `forceDispatch` / BroadcastChannel | 教学场景广播 vs 微前端宿主 |

**架构取舍**：不把 HTTP 长轮询塞进内核；**内核只认树 JSON**，网络与宿主差异在外围适配——利于单元测试和多端复用。

## 7. 小结（面试一句话）

**Slides Engine 采用「可观察组件树 + 行为注册表 + 工作区隔离」的低代码架构，通过显式事件与序列化边界把编辑内核与教学业务、跨 iframe 协议解耦，并在 Workspace 维度绑定历史与双视口交互。**
